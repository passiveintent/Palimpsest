# ADR-001: Language & Oracle

**Status:** Accepted (pre-alpha; implementation pending)

## Problem

Python + Go must match deterministically.

## Decision

Go monorepo for the OTel Collector wedge; Python feasibility harness
promoted to conformance oracle generating golden vectors Go must match
byte-exact.

## Consequences

* Most golden vectors are fully deterministic: hashing, quantisation, wire
  encoding, HKDF, and dict-root all compare byte-exact between oracle and Go.

* **BLAS-determinism asterisk (recovery vectors only):** `recovery_case1.json`
  and `recovery_watermark.json` are generated by 350 FISTA iterations over
  large matrices. BLAS backends (Windows/AVX2, Linux/OpenBLAS,
  macOS/Accelerate) accumulate floating-point rounding differently, amplifying
  tiny per-operation differences into visible low-order JSON digits. The
  *answers* are identical (same support, max value delta < 1 ULP at f64); only
  the serialised representation differs.

  Mitigation: `gen_golden.py` rounds all floats to 9 decimal places (1e-9)
  via `_canon` before serialising, and all JSON output uses `sort_keys=True`.
  This is far above the ~1e-12 numerical noise floor and well below the Go
  golden tests' 1e-4 tolerance, so it produces stable committed JSON on every
  machine without affecting any assertion that matters.

  Go golden tests for recovery already use tolerance comparisons (not
  byte-exact diff), so nothing fails — the rounding merely prevents `make
  golden` from dirtying the git tree when run on a non-Windows BLAS.
